# Opportunity: 스킬/역량 표준화 체계

**식별일:** 2025-11-16
**출처:** 보리 인터뷰 (snapshot-borry-2025-11-16.md)
**우선순위:** High
**임팩트 영역:** 채용 효율성, 팀 간 일관성, 지원자 경험

---

## 고객 문제 (Customer Problem)

### 구체적 상황
채용 담당자(보리)는 **같은 직급과 스킬에 대해 팀마다 다른 정의와 용어를 사용**하여, 매 채용마다 JD를 처음부터 작성하고 팀 간 합의에 일주일을 소비합니다.

### 실제 사례
**시니어 백엔드 개발자 채용 시:**
- **직무 레벨 기준 불일치:**
  - A팀: "경력 5년 이상이면 시니어"
  - B팀: "리딩 경험이 있어야 시니어"
- **스킬 용어 불일치:**
  - 같은 능력을 A팀은 "백엔드 API 설계", B팀은 "마이크로서비스 아키텍처"로 표현
  - 실제로는 모두 "RESTful API 개발" 스킬을 의미
- **소프트 스킬 정의 부재:**
  - "커뮤니케이션 스킬"이 발표 능력인지, 문서 작성 능력인지, 경청 능력인지 면접관마다 다름

### 고객의 목소리
> "같은 'Senior Backend Developer' 포지션인데도 JD를 작성할 때마다 처음부터 다시 써야 하고, 두 팀 간의 합의점을 찾는 데만 일주일이 걸렸어요."

> "'커뮤니케이션 스킬'이라고 하면, 어떤 면접관은 발표 능력을 보고, 어떤 면접관은 문서 작성 능력을 보고, 또 다른 면접관은 경청 능력을 봐요. 같은 말을 하는데 각자 머릿속에 그리는 그림이 다 다른 거죠."

> "'저희 회사 시니어 개발자가 뭐예요?'라고 물어보면 저도 명확하게 답을 못 해요."

## 현재 해결 방법과 한계

### 현재 시도한 방법
1. **노션 템플릿 제작**
   - JD 템플릿에 스킬 항목 사전 정의
   - 결과: 사람들이 "내 상황에 안 맞는다"며 무시
   - 한계: 경직되어 있어 유연성 부족

2. **ATS(채용 관리 시스템) 활용**
   - 스킬 태깅 기능 사용 시도
   - 결과: 스킬 자체가 표준화되어 있지 않아 무용지물
   - 한계: 사람이 수동으로 정리해야 하는데 시간 부족

3. **팀 간 합의 회의**
   - 매 채용마다 관련 팀들과 회의
   - 결과: 일주일 소요, 임시방편적 합의
   - 한계: 표준이 조직에 축적되지 않고 일회성으로 끝남

### 왜 실패했는가?
- **경직성 vs 유연성 딜레마:** 고정된 템플릿은 현실의 다양한 상황을 담지 못함
- **단순 용어 정의의 한계:** 스킬 간 관계, 레벨 간 차이, 맥락적 의미를 표현할 수 없음
- **개인 노력의 한계:** 회사 차원의 체계 없이 개인이 정리하기엔 범위가 너무 광범위

## 온톨로지 기반 솔루션 가능성

### HR 온톨로지가 해결할 수 있는 것

**1. 계층적 스킬 분류 체계**
```
기술 스킬
├─ 백엔드 개발
│  ├─ API 개발
│  │  ├─ RESTful API 설계 (isA: API 개발)
│  │  └─ GraphQL API 설계 (isA: API 개발)
│  └─ 아키텍처
│     └─ 마이크로서비스 아키텍처 (requires: API 개발)
```

**2. 레벨 정의와 요구 스킬 매핑**
```
시니어 백엔드 개발자
├─ 필수 스킬: RESTful API 개발 (숙련도: Expert)
├─ 필수 경험: 경력 5년+ OR 리딩 경험 2년+
└─ 선호 스킬: 마이크로서비스 아키텍처 (숙련도: Advanced)
```

**3. 다중 용어 지원 (동의어 관리)**
```
RESTful API 개발
├─ 동의어: "백엔드 API 설계"
├─ 동의어: "REST API 구현"
├─ 관련 개념: "마이크로서비스 아키텍처" (broader)
```

**4. 소프트 스킬의 구체화**
```
커뮤니케이션
├─ 구두 커뮤니케이션
│  ├─ 발표/프레젠테이션
│  └─ 1:1 대화
├─ 서면 커뮤니케이션
│  ├─ 기술 문서 작성
│  └─ 이메일/메시지
└─ 경청 및 이해
```

### 핵심 가치 제안
- **일관성**: 모든 팀이 같은 스킬 정의 사용
- **유연성**: 동의어, 계층 구조로 다양한 표현 수용
- **맥락화**: 스킬 간 관계, 레벨별 요구사항 명확히 정의
- **확장성**: 새로운 스킬, 직무 추가 시 기존 체계와 자동 연결

## 예상 임팩트

### 정량적 임팩트
- **JD 작성 시간 단축:** 2시간 → 30분 (75% 감소)
  - 근거: 표준 스킬/레벨 정의를 선택하고 조합하면 됨
- **팀 간 합의 시간 단축:** 1주일 → 1일 (85% 감소)
  - 근거: 공통 언어와 정의를 이미 보유
- **채용 사이클 단축:** 평균 7일 단축
  - 근거: JD 작성 및 합의 단계 시간 절감

### 정성적 임팩트
- **일관성 향상:**
  - 같은 직급/스킬에 대해 회사 전체가 동일하게 이해
  - 지원자 경험 개선 (명확한 기대치 전달)
- **지식 재사용:**
  - 과거 채용에서 정의한 스킬/레벨을 다음 채용에서 즉시 활용
  - 조직의 채용 지식이 체계적으로 축적
- **팀 간 협업 강화:**
  - 공통 용어로 소통하여 오해 감소
  - 부서 간 인재 이동 시에도 일관된 기준 적용
- **확신 있는 의사결정:**
  - 명확한 기준에 따라 포지션 정의
  - "이게 맞나?" 하는 불안감 해소

### 2차 효과
- **조직 확장성:** 팀 증가 시에도 표준 유지 가능
- **채용 데이터 분석:** 표준화된 스킬 데이터로 채용 성과 분석 가능
- **교육 프로그램 연계:** 부족한 스킬에 대한 체계적 교육 설계

## 가정 (Assumptions)

### 검증 필요한 가정
1. **사용자 학습 부담:**
   - 가정: HR 담당자가 온톨로지 기반 시스템을 1주일 내 학습 가능
   - 리스크: 복잡도가 높아 채택 실패
   - 검증 방법: 프로토타입 사용성 테스트

2. **표준 수용성:**
   - 가정: 팀들이 회사 표준을 받아들일 의향 있음
   - 리스크: "우리 팀은 다르다"며 거부
   - 검증 방법: 팀 리더 인터뷰, 파일럿 테스트

3. **초기 온톨로지 구축 비용:**
   - 가정: 기본 HR 온톨로지를 2주 내 구축 가능
   - 리스크: 도메인 복잡도로 3개월 이상 소요
   - 검증 방법: 스킬 분류 체계 프로토타입 개발

4. **다양성 vs 표준 균형:**
   - 가정: 온톨로지가 충분히 유연하여 다양한 직무/스킬 수용 가능
   - 리스크: 경직된 분류 체계로 실제 니즈 반영 실패
   - 검증 방법: 실제 JD 10개로 커버리지 테스트

## 경쟁 솔루션 분석

### 기존 시장 솔루션
1. **일반 ATS (Applicant Tracking System):**
   - 제공: 스킬 태깅 기능
   - 한계: 스킬 정의가 자유 텍스트, 표준화 없음
   - 차별점: HR 온톨로지는 스킬 간 관계, 계층, 동의어 체계 제공

2. **스킬 분류 프레임워크 (O*NET, ESCO):**
   - 제공: 포괄적 직무/스킬 분류
   - 한계: 일반적이고 광범위하여 회사 맥락 없음
   - 차별점: 회사별 커스터마이징 가능한 온톨로지

3. **HR 컨설팅:**
   - 제공: 역량 모델 설계 서비스
   - 한계: 일회성 문서, 시스템화 안 됨
   - 차별점: 실시간 업데이트되고 시스템에 통합된 살아있는 온톨로지
