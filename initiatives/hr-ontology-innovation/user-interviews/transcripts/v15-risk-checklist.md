# V1.5 우려사항 및 오버스펙 체크리스트

**작성일:** 2025-11-27
**목적:** Borry 인터뷰 시 실무적 관점의 리스크 체크
**사용:** 인터뷰 중 또는 인터뷰 후 검토

---

## 🚨 Critical 우려사항 (블로커 가능성)

### 1. 데이터 제공 불가

**우려:** 과거 채용 데이터를 제공할 수 없음

**체크리스트:**
- [ ] 법적/윤리적 이슈 (개인정보 보호법 위반)
- [ ] 데이터 소유권 문제 (회사 승인 필요, 받기 어려움)
- [ ] 데이터가 너무 파편화 (노션, 스프레드시트, 이메일 등 분산)
- [ ] 익명화 프로세스가 너무 복잡 (몇 주 소요)

**질문:**
> "과거 채용 데이터 5-10건 제공이 불가능한 이유가 있을까요?"

**완화 전략:**
- 3건만 제공하고 나머지 가상 데이터
- 특정 포지션 (Backend Senior) 1개만 집중
- V1.5 전체 보류, V1에 집중

**블로커 여부 판단:**
- ✅ 3건이라도 제공 가능 → 진행
- ❌ 전혀 제공 불가능 → **V1.5 보류**

---

### 2. 유용성 의심

**우려:** DCQ 정보가 실제로 도움이 안 됨

**체크리스트:**
- [ ] "이런 정보 몰라도 채용 잘 하고 있다"
- [ ] "데이터 기반 의사결정보다 직감이 더 중요하다"
- [ ] "과거 패턴이 미래를 예측할 수 없다"
- [ ] "우리 회사는 채용 데이터가 너무 적어서 (연 10건 미만) 의미 없다"

**질문:**
> "지난 채용에서 '과거 합격자 패턴'을 알았으면 도움이 됐을 상황이 있었나요?"

**완화 전략:**
- Top 3 DCQ만 선택 (Borry가 유용하다고 느끼는 것만)
- "실험" 마인드셋 (3개월 써보고 판단)

**블로커 여부 판단:**
- ✅ 최소 3개 DCQ는 유용하다 → 진행 (축소 범위)
- ❌ 전혀 유용하지 않다 → **V1.5 보류**

---

### 3. 문화적 저항

**우려:** 면접관들이 평가 패턴 공개를 거부

**체크리스트:**
- [ ] "제임스가 '나는 엄격하다'는 데이터 보면 기분 나빠할 것"
- [ ] "평가 점수 공개하면 팀 갈등 생길 것"
- [ ] "면접관들이 데이터 기반 피드백을 받아들이지 않을 것"

**질문:**
> "면접관별 평가 패턴을 공유하면 민감한 이슈가 될까요?"

**완화 전략:**
- 면접관 익명화 ("Interviewer A")
- 면접관에게는 비공개, HR만 확인
- Positive framing ("엄격"이 아닌 "높은 기준")

**블로커 여부 판단:**
- ✅ 익명화/비공개로 해결 가능 → 진행
- ❌ 어떤 방식으로도 불가능 → **DCQ-04 제거**

---

## ⚠️ High 우려사항 (범위 축소 가능성)

### 4. 복잡도 과다

**우려:** 동적 개념 9개는 너무 많다

**체크리스트:**
- [ ] "9개는 이해하기 어렵다. 3-5개가 적당"
- [ ] "시스템 사용법 학습에 1주일 이상 걸릴 것 같다"
- [ ] "데이터 입력이 너무 번거로울 것 같다"

**질문:**
> "9개 동적 개념이 너무 많다고 느껴지나요? 몇 개가 적당할까요?"

**완화 전략:**
- P0만 먼저 (7개 → 5개)
- UI로 복잡도 숨김
- 단계적 공개 (처음엔 3개만 → 점진적 확장)

**대응:**
- Borry 제안대로 개수 축소
- V1.5.1 (간단) → V1.5.2 (확장) 단계화

---

### 5. 데이터 품질 낮음

**우려:** 과거 데이터가 불완전/부정확

**체크리스트:**
- [ ] "탈락 사유를 명확히 기록하지 않는다"
- [ ] "면접 평가 점수가 일관되지 않다"
- [ ] "과거 데이터에 누락이 많다"
- [ ] "데이터 입력 방식이 면접관마다 다르다"

**질문:**
> "과거 채용 데이터 품질이 어느 정도인가요? 신뢰할 만한가요?"

**완화 전략:**
- V1.5 시작하면서 데이터 품질 개선 (앞으로는 일관되게 입력)
- 불완전한 데이터는 제외하고 품질 높은 것만 사용
- "최소 10건 이상 모이면 분석 시작" 기준 설정

**대응:**
- 데이터 품질 개선 프로세스 병행
- 초기엔 경고 메시지 ("데이터 부족, 참고만 하세요")

---

### 6. 시간 투입 부담

**우려:** Borry의 시간 투입 9시간이 부담됨

**체크리스트:**
- [ ] "9시간은 너무 많다. 3-5시간이 한계"
- [ ] "샘플 데이터 수집 4시간이 제일 부담"
- [ ] "지금은 바빠서 나중에 하고 싶다"

**질문:**
> "Week 5-7 동안 총 9시간 정도 필요한데, 부담되시나요?"

**완화 전략:**
- 샘플 데이터 수집 지원 (Terry가 직접 도움)
- 시간 축소 (9시간 → 5시간)
  - 검토 세션: 2시간 → 1시간
  - 샘플 데이터: 4시간 → 2시간 (3건만)
- 일정 조율 (바쁘지 않은 시기로)

**대응:**
- Borry 가능 시간에 맞춰 조정
- 우선순위 낮은 작업 제거

## 📉 오버스펙 체크 (제거 가능)

### 7. DCQ 8개가 너무 많음

**체크:**
- [ ] DCQ-02 (Competency 평가 시간): "면접 시간은 정해져 있어서 최적화 불필요"
- [ ] DCQ-03 (합격률): "합격률은 시장 상황에 따라 바뀌어서 의미 없음"
- [ ] DCQ-07 (탈락 사유): "탈락 사유 기록을 잘 안 해서 데이터 품질 낮음"

**질문:**
> "8개 DCQ 중에서 '이건 과하다, 불필요하다'고 느껴지는 것이 있나요?"

**대응:**
- 불필요한 DCQ 제거
- Top 3-5개만 V1.5에 포함
- 나머지는 V2 또는 보류
