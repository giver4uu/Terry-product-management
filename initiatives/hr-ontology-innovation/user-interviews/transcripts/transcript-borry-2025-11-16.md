## 인터뷰 내용

### 도입: 현재 채용 프로세스에 대해

**Terry:** 보리님, 오늘은 현재 채용 업무에서 겪고 계신 어려움에 대해 구체적으로 듣고 싶습니다. 특히 최근에 진행하신 채용 프로세스 중에서 기억에 남는 사례가 있으신가요?

**보리:** 네, 사실 지난달에 시니어 백엔드 개발자를 채용하면서 정말 답답했던 경험이 있어요. 저희가 백엔드 팀이 2개가 있는데, 각 팀마다 '시니어' 개발자에 대한 기준이 완전히 달라요.

**Terry:** 구체적으로 어떻게 달랐나요?

**보리:** A팀은 경력 5년 이상이면 시니어라고 생각하고, B팀은 리딩 경험이 있어야 시니어라고 봐요. 그래서 같은 'Senior Backend Developer' 포지션인데도 JD를 작성할 때마다 처음부터 다시 써야 하고, 두 팀 간의 합의점을 찾는 데만 일주일이 걸렸어요.

### 직무-스킬 정의의 비표준화 문제

**Terry:** JD 작성 외에 다른 어려움도 있으셨나요?

**보리:** 있죠. 스킬 정의가 제일 큰 문제예요. 예를 들어, "RESTful API 개발" 능력이 필요하다고 하면, A팀은 그걸 "백엔드 API 설계"라고 표현하고, B팀은 "마이크로서비스 아키텍처"라고 써요. 결국 같은 스킬을 말하는 건데 표현이 다 달라요.

**Terry:** 그게 실무에서 어떤 문제를 만들었나요?

**보리:** 지난번 채용 때요, 저희가 노션에 면접 질문을 30개 정도 모아뒀거든요. 그런데 막상 쓰려고 보니까 이게 언제, 누가, 어떤 포지션을 위해서 만든 질문인지 알 수가 없는 거예요. "리더십"을 평가하는 질문이 5개가 있는데, 어떤 건 팀 관리 경험을 물어보고, 어떤 건 의사결정 스타일을 물어보고... 맥락이 전혀 없으니까 매번 처음부터 다시 만들어야 해요. 그러면 2시간은 훌쩍 가버리죠.

**Terry:** 소프트 스킬은 어떤가요?

**보리:** 그건 더 심해요. "커뮤니케이션 스킬"이라고 하면, 어떤 면접관은 발표 능력을 보고, 어떤 면접관은 문서 작성 능력을 보고, 또 다른 면접관은 경청 능력을 봐요. 같은 말을 하는데 각자 머릿속에 그리는 그림이 다 다른 거죠.

### 평가 기준의 일관성 부족

**Terry:** 평가 단계에서는 어떠셨나요?

**보리:** (한숨을 쉬며) 이게 제일 좌절스러워요. 지난번 시니어 백엔드 개발자 채용 때, 면접관이 3명이었어요. 모두 "커뮤니케이션"을 평가 항목으로 뽑았는데, 결과가 완전히 달랐어요.

**Terry:** 구체적으로 어떻게 달랐죠?

**보리:** 제임스(개발 리드)는 5점을 줬어요. "기술 내용을 명확하게 설명했다"는 이유로요. 그런데 마케팅 팀장은 3점을 줬어요. "비개발자에게 설명하는 능력이 부족하다"고 했고요. 또 다른 개발자는 4점을 줬는데, 근거가 "질문에 잘 답변했다" 정도였어요.

**Terry:** 그럼 최종 결정은 어떻게 하셨어요?

**보리:** 회의를 한 시간 동안 했어요. 각자 "5점"이 뭔지, "3점"이 뭔지에 대한 정의가 전혀 없으니까 논의가 계속 엇나가더라고요. 결국 제임스의 의견을 따랐는데, 이게 맞는 결정인지 확신이 없었어요. 루브릭에는 점수만 있고, 각 점수가 구체적으로 뭘 의미하는지 명확한 기준이 없으니까요.

### 채용 지식의 재사용 불가

**Terry:** 아까 노션에 면접 질문이 있다고 하셨는데, 그게 도움이 안 되나요?

**보리:** 거의 안 돼요. 질문만 있지, 맥락이 없어요. 예를 들어, "팀 내 갈등을 어떻게 해결했나요?"라는 질문이 있는데, 이게:
- 언제 만들어진 질문인지
- 어떤 포지션(주니어? 시니어? 리드?)을 위한 건지
- 누가 만들었고, 왜 만들었는지
- 과거에 이 질문으로 뽑은 사람이 실제로 어땠는지

이런 정보가 하나도 없어요. 그래서 결국 안 쓰고 매번 새로 만들게 돼요.

**Terry:** 과거 채용 데이터는 어떻게 활용하시나요?

**보리:** (쓴웃음을 지으며) 활용을 못 해요. 작년에 백엔드 개발자 5명을 뽑았는데, 그때 합격자들이 어떤 스킬을 가지고 있었는지, 어떤 질문에 어떻게 답했는지 찾기가 너무 어려워요. 노션 페이지가 100개가 넘는데 검색도 잘 안 되고... 그냥 기억에 의존하게 되죠.

### 조직 간 노하우 공유 장벽

**Terry:** 팀 간 채용 정보 공유는 어떻게 하시나요?

**보리:** 거의 안 돼요. 백엔드 팀이 2개 있는데, 각자 따로따로 채용해요. A팀이 좋은 면접 질문을 만들어도 B팀은 몰라요. 공유할 시스템이 없으니까요.

**Terry:** 제임스님의 노하우도 말씀하셨는데...

**보리:** 네, 제임스는 채용을 정말 잘해요. 10년 경력이 있고, 면접 때 질문하는 방식도 체계적이에요. 그런데 그게 제임스의 머릿속에만 있어요. 문서화가 안 되어 있으니까 다른 면접관들은 배울 수가 없죠. 제임스가 바빠서 면접에 못 오면, 퀄리티가 확 떨어져요.

**Terry:** 다른 팀들도 비슷한 문제가 있나요?

**보리:** 있죠. 프론트엔드 팀, DevOps 팀, 데이터 팀... 모두 각자의 방식으로 채용하는데, 직무 레벨 정의가 팀마다 달라요. 그래서 지원자 입장에서는 혼란스러워요. "저희 회사 시니어 개발자가 뭐예요?"라고 물어보면 저도 명확하게 답을 못 해요.

### 감정과 임팩트

**Terry:** 이런 문제들이 보리님께 어떤 영향을 주나요?

**보리:** (잠시 생각하다가) 솔직히... 무력감이 들어요. 채용이 회사의 미래를 좌우하는데, 매번 제대로 하고 있는지 확신이 없어요. 그리고 시간이 너무 아까워요. JD 쓰는 데 2시간, 면접 질문 만드는 데 2시간, 팀 간 합의하는 데 또 2시간... 일주일에 하루는 이런 비효율적인 일에 쓰는 것 같아요.

**Terry:** 가장 해결하고 싶은 문제는 뭔가요?

**보리:** 세 가지예요. 첫째는 스킬이나 역량을 표준화하고 싶어요. "시니어 백엔드 개발자"라고 하면 누구나 같은 기준을 떠올릴 수 있게요. 둘째는 평가를 일관되게 하고 싶어요. 면접관마다 다른 기준으로 평가하는 게 아니라, 명확한 루브릭이 있었으면 좋겠어요. 셋째는 과거 채용 데이터를 재사용하고 싶어요. 좋은 질문, 좋은 후보자 프로필, 이런 걸 쌓아서 계속 개선할 수 있으면 얼마나 좋을까요.

### 현재 해결 시도

**Terry:** 이 문제를 해결하기 위해 시도해보신 것이 있나요?

**보리:** 네, 노션에 템플릿을 만들어봤어요. JD 템플릿, 면접 질문 템플릿, 평가표 템플릿... 그런데 사람들이 안 써요. 귀찮아하거든요. 템플릿이 유연하지 않으니까 "내 상황에 안 맞는다"고 하면서 무시해버려요.

**Terry:** 다른 도구는 고려해보셨나요?

**보리:** 채용 관리 시스템(ATS)도 써봤는데, 거기도 마찬가지예요. 스킬 태깅 같은 기능이 있긴 한데, 스킬 자체가 표준화되어 있지 않으니까 의미가 없어요. 결국 사람이 수동으로 정리해야 하는데, 그럴 시간이 어디 있어요?

### 이상적인 해결책

**Terry:** 만약 마법처럼 이 문제가 해결된다면, 어떤 모습일까요?

**보리:** (눈을 반짝이며) 상상만 해도 좋네요. 제가 "백엔드 개발자"라고 입력하면, 시스템이 자동으로:
1. 표준화된 스킬 리스트를 제안해주고
2. 그 스킬에 맞는 면접 질문을 추천해주고
3. 과거 합격자들이 어떤 답변을 했는지 보여주고
4. 다른 팀의 베스트 프랙티스도 공유해주고
5. 평가할 때는 각 점수가 정확히 뭘 의미하는지 가이드를 줘요.

그리고 제일 중요한 건, 이게 한 번 쓰고 버리는 게 아니라 계속 쌓여서 더 똑똑해지는 거예요. 채용할 때마다 데이터가 쌓이고, 그게 다음 채용에 도움이 되는... 그런 선순환이요.

**Terry:** 그런 시스템이 있다면 얼마나 도움이 될까요?

**보리:** 엄청나죠. 제 시간의 50%는 아낄 수 있을 것 같고, 더 중요한 건 채용 퀄리티가 올라갈 거예요. 지금은 운에 맡기는 느낌인데, 체계가 잡히면 일관되게 좋은 사람을 뽑을 수 있을 것 같아요.

## 주요 관찰사항

### 행동 패턴
- 매 채용마다 JD와 면접 질문을 처음부터 작성 (2시간씩 소요)
- 노션에 자료가 있어도 맥락 부족으로 재사용하지 않음
- 팀 간 합의를 위한 회의에 많은 시간 소비 (일주일 이상)
- 평가 회의에서 점수 기준 논쟁으로 1시간 소요

### 감정 반응
- **좌절감**: 평가 기준 불일치로 최종 결정에 확신 없음
- **무력감**: 제대로 하고 있는지 확신 없음
- **시간 아까움**: 비효율적인 작업에 일주일에 하루 소비
- **기대감**: 문제 해결에 대한 명확한 비전과 기대

### 문제 해결 시도
- 노션 템플릿 제작 → 실패 (사용자들이 유연성 부족으로 무시)
- ATS 도입 → 실패 (스킬 표준화 부재로 효과 없음)
- 수동 정리 시도 → 실패 (시간 부족)

### 핵심 니즈
1. 직무/스킬 표준화 체계
2. 명확한 평가 루브릭
3. 채용 지식의 재사용과 축적
4. 팀 간 노하우 공유
5. 학습하고 개선되는 시스템

## 인터뷰 후 메모

- 문제의 근본 원인은 "지식의 파편화"와 "표준의 부재"
- 단순한 도구 문제가 아니라 "맥락 있는 지식 체계" 부재
- HR 온톨로지가 해결할 수 있는 명확한 문제 영역 확인

**검증 필요:**
- 다른 회사 HR 담당자들도 유사한 문제를 겪는가?
- 회사 규모별로 문제의 심각도가 다른가?
- 온톨로지 기반 솔루션에 대한 학습 비용은 감당 가능한가?

**후속 질문:**
- 제임스님의 면접 노하우를 구체적으로 들어볼 수 있는가?
- 실제 면접 평가표를 공유받을 수 있는가?
- 과거 합격자 데이터의 현재 보관 형태는?
