**질문:**
> "지난 채용에서 '어느 단계가 오래 걸렸는지' 알았으면 도움이 됐을 상황이 있었나요?"

**확인 포인트:**

#### DCQ-04: 면접관 평가 패턴

**질문:**
> "면접관마다 평가 기준이 다르다는 걸 느낀 적 있나요? 예를 들어 제임스가 유독 엄격하다든지?"

**확인 포인트:**
- [ ] 실제로 면접관 간 평가 차이를 경험했는가?
- [ ] "제임스는 Communication 평가 시 0.4점 더 엄격" 같은 정보가 유용한가?

**우려사항 체크:**
- [ ] "면접관 패턴을 공개하면 민감한 이슈가 될 수 있다"
- [ ] "Calibration 하려면 추가 미팅 필요 → 시간 부담"
- [ ] "면접관이 이 데이터를 거부감 없이 받아들일까?"

**우려사항 체크:**
- [ ] "합격자가 너무 적으면 (5명 미만) 통계적으로 의미 없다"

### Part 3: 샘플 데이터 제공 가능성 (10분)

**필요한 데이터:**
```
1건당 필요 정보:
  - 포지션: "Senior Backend Engineer - Payments"
  - 지원자 기본 정보 (익명화)
  - 면접 단계별 일정 (리드타임 계산용)
  - 면접 평가 기록
    - 면접관 이름 (익명 가능: "Interviewer A")
    - Competency별 점수 (PostgreSQL: 4.5/5)
  - 최종 결정 (합격/불합격 + 사유)
```

**확인 포인트:**
- [ ] 과거 데이터가 어떤 형태로 저장되어 있는가? (노션? 스프레드시트? ATS?)

- [ ] "데이터가 너무 파편화되어 있어서 모으기 어렵다"
- [ ] "법적/윤리적 이슈로 제공 불가능하다"
- [ ] "익명화에 시간이 너무 많이 걸린다"
- [ ] "과거 데이터 품질이 낮다 (누락, 불일치)"

**대안 논의:**
- 최소 3건만 제공하고 나머지는 가상 데이터?
- 최근 3개월 데이터만 우선 제공?
