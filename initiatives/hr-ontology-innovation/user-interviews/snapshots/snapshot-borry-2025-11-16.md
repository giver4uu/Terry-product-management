  - 매 채용마다 JD와 면접 질문을 처음부터 작성 (2시간씩 소요)

- **상황:** 백엔드 팀 2개를 포함한 다수 팀의 채용 총괄, 각 팀마다 다른 채용 기준으로 혼란

## 스토리 요약

### 스토리 1: 시니어 백엔드 개발자 채용의 좌절

**맥락:** 지난달 시니어 백엔드 개발자 채용 프로세스. 백엔드 팀이 2개 있는 상황에서 같은 직급의 포지션을 채용해야 했음.

**발생한 일:**
1. JD 작성 단계에서 A팀과 B팀의 '시니어' 기준이 완전히 다름을 발견
   - A팀: 경력 5년 이상이면 시니어
   - B팀: 리딩 경험이 있어야 시니어
2. 스킬 정의도 팀마다 다른 용어 사용
   - A팀: "백엔드 API 설계"
   - B팀: "마이크로서비스 아키텍처"
   - 실제로는 같은 스킬인 "RESTful API 개발"을 의미
3. 두 팀 간 합의점을 찾기 위해 일주일 동안 논의
4. 최종 JD 완성까지 처음부터 다시 작성

**결과:** 일주일의 시간 소비, 팀 간 합의는 했지만 표준화된 기준 없이 임시방편으로 해결

**주요 순간:**
- **좌절의 순간**: "같은 직급인데 왜 매번 처음부터 정의해야 하지?"
- **비효율 인식**: JD 작성에만 2시간, 팀 간 조율에 일주일
- **근본 문제 파악**: 스킬과 직무 레벨에 대한 회사 차원의 표준이 없음

### 스토리 2: 면접 평가의 혼란

**맥락:** 시니어 백엔드 개발자 후보자의 최종 면접. 면접관 3명 (개발 리드 제임스, 마케팅 팀장, 다른 개발자)이 평가에 참여.

**발생한 일:**
1. 모든 면접관이 "커뮤니케이션" 항목을 평가하기로 합의
2. 면접 후 평가 결과 공유:
   - 제임스: 5점 - "기술 내용을 명확하게 설명했다"
   - 마케팅 팀장: 3점 - "비개발자에게 설명하는 능력이 부족하다"
   - 다른 개발자: 4점 - "질문에 잘 답변했다"
3. 점수 차이의 원인을 논의하는 회의 1시간 진행
4. 각자 "5점", "3점"의 의미가 다름을 발견
5. 명확한 기준 없이 제임스의 의견을 따르기로 결정

**결과:** 최종 결정은 했지만 확신 없음. 루브릭에 점수별 명확한 기준이 없어 논의가 계속 엇나감.

**주요 순간:**
- **혼란의 순간**: "같은 '커뮤니케이션'을 봤는데 왜 점수가 5점, 3점, 4점으로 다 다르지?"
- **무력감**: "이게 맞는 결정인지 확신이 없었어요"
- **시스템 부재 인식**: 각 점수가 구체적으로 뭘 의미하는지 정의 없음

### 스토리 3: 노션 자료의 무용지물

**맥락:** 새로운 채용을 준비하면서 과거에 축적한 노션 자료를 활용하려던 시도.

**발생한 일:**
1. 노션에 면접 질문 30개가 모아져 있음을 확인
2. "팀 내 갈등을 어떻게 해결했나요?" 같은 질문 발견
3. 그러나 질문의 맥락 정보가 전혀 없음:
   - 언제 만들어진 질문인지
   - 어떤 포지션(주니어? 시니어? 리드?)용인지
   - 누가 왜 만들었는지
   - 과거 이 질문으로 뽑은 사람이 어땠는지
4. 결국 질문을 사용하지 않고 처음부터 새로 작성
5. 2시간 소요

**결과:** 축적된 지식이 있지만 활용 불가. 매번 같은 작업을 반복.

**주요 순간:**
- **시간 낭비 인식**: "거의 안 돼요. 질문만 있지, 맥락이 없어요"
- **반복 작업의 피로**: "그래서 결국 안 쓰고 매번 새로 만들게 돼요"
- **데이터 찾기의 어려움**: "노션 페이지가 100개가 넘는데 검색도 잘 안 되고..."

---

### 스토리 4: 제임스의 노하우는 제임스 안에만

**맥락:** 10년 경력의 개발 리드 제임스가 만든 우수한 채용 프로세스가 있지만, 다른 팀원들은 접근 불가.

**발생한 일:**
1. 제임스는 체계적인 면접 방식으로 항상 좋은 결과 도출
2. 그러나 그의 노하우는 문서화되어 있지 않음
3. 다른 면접관들은 제임스의 방법을 배울 수 없음
4. 제임스가 바빠서 면접에 참여하지 못하는 경우:
   - 면접 퀄리티가 눈에 띄게 하락
   - 다른 팀원들은 자기 방식대로 진행할 수밖에 없음
5. 백엔드 A팀과 B팀이 따로 채용하며 노하우 공유 없음

**결과:** 개인의 우수 사례가 조직의 자산으로 축적되지 못함. 채용 퀄리티가 담당자에 따라 들쑥날쑥.

**주요 순간:**
- **인적 의존성 인식**: "제임스가 바빠서 면접에 못 오면, 퀄리티가 확 떨어져요"
- **지식 공유 장벽**: "A팀이 좋은 면접 질문을 만들어도 B팀은 몰라요. 공유할 시스템이 없으니까요"
- **표준 부재**: "'저희 회사 시니어 개발자가 뭐예요?'라고 물어보면 저도 명확하게 답을 못 해요"

### 스토리 5: 템플릿의 실패

**맥락:** 문제를 해결하기 위해 노션에 채용 템플릿을 만들어 배포한 시도.

**발생한 일:**
1. JD 템플릿, 면접 질문 템플릿, 평가표 템플릿 제작
2. 팀원들에게 사용 권장
3. 사람들이 템플릿을 사용하지 않음
4. 이유: "내 상황에 안 맞는다", "너무 경직되어 있다"
5. 결국 템플릿 무시하고 각자 방식대로 진행

**결과:** 템플릿 제작에 투입한 시간 낭비. 표준화 실패.

**주요 순간:**
- **해결 시도**: 체계를 만들려는 노력
- **실패 원인 파악**: "템플릿이 유연하지 않으니까"
- **더 깊은 문제 인식**: 단순 템플릿이 아니라 맥락 있는 체계가 필요

## 경험 맵

**범위:** 시니어 백엔드 개발자 채용 프로세스 전체 (JD 작성 → 면접 → 평가 → 최종 결정)

**목표:** 회사에 맞는 우수한 시니어 개발자를 일관되고 효율적으로 채용

**여정 단계:**

### 1. JD 작성 단계
- **행동:**
  - 백엔드 A팀, B팀과 시니어 개발자 기준 논의
  - "경력 5년" vs "리딩 경험" 기준 충돌
  - 스킬 정의 차이 조율 ("RESTful API" vs "백엔드 API 설계")
  - 일주일간 이메일, 회의로 합의점 모색
  - 최종 JD 작성에 2시간 소요
- **생각/감정:**
  - "왜 같은 직급인데 매번 처음부터 정의해야 하지?"
  - "이 시간에 다른 일을 했으면..."
  - 좌절감과 비효율성에 대한 인식
- **페인 포인트:**
  - 직무 레벨과 스킬에 대한 회사 표준 부재
  - 팀 간 용어 불일치
  - 합의에 일주일 소요
- **도구/리소스:**
  - 노션 (과거 JD 참조 시도했으나 맥락 없어 사용 불가)
  - 이메일, Slack 회의

### 2. 면접 질문 준비 단계
- **행동:**
  - 노션에서 과거 면접 질문 30개 검색
  - "팀 내 갈등 해결" 같은 질문 발견
  - 질문의 맥락(누가, 언제, 어떤 포지션용) 정보 부재 확인
  - 사용 포기하고 새로 작성
  - 2시간 소요
- **생각/감정:**
  - "이걸 왜 모아뒀지? 쓸 수가 없잖아"
  - "매번 같은 걸 또 만들어야 한다니..."
  - 반복 작업에 대한 피로감
- **페인 포인트:**
  - 축적된 자료의 맥락 정보 부재
  - 검색 어려움 (노션 페이지 100개+)
  - 재사용 불가능한 지식 베이스
- **도구/리소스:**
  - 노션 (질문 저장소이지만 비효율적)
  - 제임스의 암묵지 (접근 불가)

### 3. 면접 실행 단계
- **행동:**
  - 면접관 3명 배정 (제임스, 마케팅 팀장, 개발자)
  - 각자 "커뮤니케이션" 평가하기로 합의
  - 면접 진행
  - 후보자는 기술 내용 설명, 질문 답변
- **생각/감정:**
  - "이번 후보는 괜찮은 것 같은데..."
  - 각자 다른 관점에서 후보자 관찰
- **페인 포인트:**
  - "커뮤니케이션"의 정의가 면접관마다 다름
  - 명확한 평가 기준 사전 합의 부재
- **도구/리소스:**
  - 엑셀 기반 평가표 (점수만 있고 기준 없음)

### 4. 평가 및 의사결정 단계
- **행동:**
  - 평가 결과 공유: 5점, 3점, 4점으로 상이
  - 1시간 동안 회의 진행
  - 각자 점수 기준 설명:
    - 제임스: "기술 설명 명확함"
    - 마케팅 팀장: "비개발자 대상 설명 부족"
    - 개발자: "질문 답변 잘함"
  - 점수의 의미에 대한 정의가 없어 논의 엇갈림
  - 최종적으로 제임스 의견 채택
- **생각/감정:**
  - "5점이 뭐고 3점이 뭐야? 기준이 뭐지?"
  - "이게 맞는 결정일까?"
  - 무력감, 확신 부족
- **페인 포인트:**
  - 루브릭의 점수별 명확한 정의 부재
  - 면접관 간 평가 기준 불일치
  - 최종 결정의 확신 부족
- **도구/리소스:**
  - 회의 (비효율적인 합의 과정)
  - 선임자(제임스)의 경험에 의존

### 5. 사후 반성 및 개선 시도 단계
- **행동:**
  - 템플릿 제작 시도 (JD, 면접 질문, 평가표)
  - 팀원들에게 배포
  - 사용률 모니터링
  - 사람들이 "유연하지 않다" 이유로 무시
  - 포기
- **생각/감정:**
  - "뭔가 체계를 만들어야 하는데..."
  - "또 실패했네"
  - "근본적인 해결책이 뭘까?"
- **페인 포인트:**
  - 경직된 템플릿은 현실에 맞지 않음
  - 단순 템플릿으론 문제 해결 안 됨
  - 더 근본적인 체계 필요
- **도구/리소스:**
  - 노션 템플릿 (실패)
  - ATS (스킬 표준화 부재로 효과 없음)

## 기회

- **직무-스킬 표준화**: 회사 차원에서 직무 레벨(주니어/시니어/리드)과 스킬(기술/소프트 스킬)에 대한 표준 정의가 필요. 팀마다 다른 용어와 기준을 사용하지 않고, 누구나 동일하게 이해할 수 있는 체계를 원함.

- **평가 루브릭 명확화**: 면접 평가 시 점수(1-5점)가 각각 무엇을 의미하는지 구체적인 기준 필요. "커뮤니케이션 5점"이 어떤 상태인지 모든 면접관이 동일하게 이해하고 평가할 수 있기를 원함.

- **채용 지식의 맥락화**: 면접 질문, JD, 평가 방법 등을 단순히 저장하는 것이 아니라, 언제/누가/왜/어떤 포지션용으로 만들었는지 맥락과 함께 저장하고 재사용할 수 있기를 원함.

- **조직 지식 공유**: 개인(제임스)이나 특정 팀의 우수 채용 노하우를 조직 전체가 활용할 수 있는 체계 필요. 팀 간 장벽 없이 베스트 프랙티스를 공유하고 학습할 수 있기를 원함.

- **학습하는 채용 시스템**: 채용을 진행할 때마다 데이터가 쌓이고, 그것이 다음 채용에 자동으로 도움이 되는 선순환 체계를 원함. 과거 합격자 프로필, 좋았던 질문, 평가 패턴 등이 누적되어 시스템이 점점 똑똑해지기를 기대.

## 인사이트

- **지식의 파편화**: 채용 관련 지식(질문, JD, 평가 기준)은 존재하지만, 맥락 정보가 없어 재사용이 불가능함. 단순 저장이 아니라 "맥락 있는 지식 체계"가 필요함을 시사.

- **표준의 부재가 비효율의 근본 원인**: 스킬, 직무 레벨, 평가 기준에 대한 회사 차원의 표준이 없어서 매번 합의에 막대한 시간 소비. 표준화가 핵심 해결책.

- **암묵지의 명시화 필요**: 제임스 같은 전문가의 노하우가 개인 안에만 존재. 우수 사례를 조직 자산으로 전환하는 메커니즘 필요.

- **템플릿의 한계**: 경직된 템플릿은 현실의 다양성을 담지 못함. 유연하면서도 일관성 있는 체계, 즉 "적응형 표준"이 필요함.

- **확신의 부재가 스트레스**: 단순히 시간이 걸리는 게 문제가 아니라, 최종 결정에 확신을 갖지 못하는 것이 더 큰 정서적 부담. 데이터 기반의 명확한 의사결정 지원이 핵심 가치.

- **HR 온톨로지의 적합성**: 보리가 설명한 모든 문제(표준화, 맥락, 공유, 학습)는 온톨로지(개념 간 관계를 정의하는 지식 체계)로 해결 가능한 영역. 기술-문제 간 명확한 핏 존재.
