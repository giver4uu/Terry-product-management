# PRD: AI 면접 노트테이커

## Introduction/Overview

AI 면접 노트테이커는 라운드의 Conversational ATS에 통합되는 혁신적인 기능으로, 면접관이 메모 작성 부담 없이 온전히 지원자와의 대화에 집중할 수 있도록 지원합니다. 화상 면접을 자동으로 녹음하고 실시간으로 전사하며, AI 기반 분석을 통해 구조화된 평가 리포트를 생성합니다.

**핵심 문제:**
- 면접관이 필기에 집중하느라 지원자와의 대화 흐름을 놓치고, 미묘한 표현과 태도를 관찰하지 못함
- 면접 후 기억에 의존한 주관적 피드백 작성으로 인한 시간 소요 (평균 28분) 및 낮은 제출률 (72%)
- 객관적 증거 부족으로 인한 디브리프 미팅의 비효율성과 주관적 평가

**해결 방안:**
- AI가 면접 전 과정을 녹음/전사하여 면접관이 대화에만 집중할 수 있는 환경 조성
- 면접 종료 즉시 AI 요약 리포트 제공으로 피드백 작성 시간 단축
- 실제 대화 내용을 기반으로 한 객관적이고 증거 기반의 평가 지원
- 온톨로지 지식 그래프를 통한 구조화된 역량 분석 및 조직 채용 지식 축적

## Goals

1. **면접 경험 향상**: 면접관이 필기 부담 없이 지원자와의 대화에 집중하여 면접 품질을 향상시킨다
2. **업무 효율성 향상**: 피드백 작성 시간을 64% 단축 (평균 28분 → 10분)하고 제출률을 95%로 향상시킨다
3. **객관적 평가 지원**: 실제 면접 대화 내용을 기반으로 한 증거 기반 평가를 가능하게 한다
4. **면접 품질 일관성 확보**: AI 실시간 질문 제안을 통해 면접관 경험 수준과 무관하게 일관된 고품질 면접을 수행한다
5. **디브리프 효율성 향상**: 객관적 면접 기록을 활용하여 디브리프 완료 시간을 52% 단축한다
6. **채용 데이터 자산화**: 면접 대화를 온톨로지 지식 그래프로 구조화하여 장기적인 조직 채용 역량을 구축한다

## User Stories

### 면접관 (주요 사용자)
1. **면접 집중**: 면접관으로서, 나는 메모 작성 없이 지원자의 눈을 바라보며 깊이있는 대화를 나누고 싶다, 지원자의 진짜 모습을 파악하기 위해
2. **빠른 피드백 작성**: 면접관으로서, 나는 면접 종료 즉시 AI 요약 리포트를 받고 싶다, 기억에 의존하지 않고 5분 내로 정확한 피드백을 작성하기 위해
3. **실시간 가이던스**: 면접관으로서, 나는 지원자 답변을 듣는 중에 추가로 물어볼 질문 제안을 받고 싶다, 역량을 더 깊이 검증하기 위해
4. **객관적 평가 근거**: 면접관으로서, 나는 실제 면접 대화 내용을 근거로 평가를 작성하고 싶다, 주관적 느낌이 아닌 사실 기반으로 평가하기 위해

### 채용 담당자
5. **팀 협업 강화**: 채용 담당자로서, 나는 면접 녹음과 요약을 팀원들과 쉽게 공유하고 싶다, 참석하지 못한 팀원도 면접 내용을 이해하고 의견을 제시할 수 있도록
6. **디브리프 효율화**: 채용 담당자로서, 나는 디브리프 미팅에서 각 지원자의 면접 녹음을 함께 검토하고 싶다, 증거 기반으로 빠르게 합의에 도달하기 위해
7. **Slack 통합**: 채용 담당자로서, 나는 Slack에서 "김철수 면접 요약해줘"라고 물어보고 싶다, 별도 UI로 이동하지 않고 빠르게 정보를 확인하기 위해

### 지원자
8. **존중받는 경험**: 지원자로서, 나는 면접관이 노트북만 보지 않고 나의 눈을 보며 경청해주기를 원한다, 진정으로 존중받는다고 느끼기 위해
9. **투명한 프로세스**: 지원자로서, 나는 면접이 녹음된다는 것을 사전에 명확히 알고 동의 여부를 선택하고 싶다, 내 개인정보를 통제하기 위해

## Functional Requirements

### 1. 녹음 및 전사 기능

#### 1.1 화상회의 자동 참여
- 시스템은 Zoom 또는 Google Meet 면접 일정에 AI 노트테이커를 자동으로 참여시켜야 한다
- AI 노트테이커는 면접 시작 시간 1분 전에 화상회의에 입장해야 한다
- AI 노트테이커는 화상회의 참여 시 명확한 이름 (예: "Round AI Notetaker")으로 표시되어야 한다
- 면접관은 대시보드에서 AI 노트테이커 자동 참여를 활성화/비활성화할 수 있어야 한다

#### 1.2 지원자 동의 프로세스
- 시스템은 AI 노트테이커가 화상회의에 입장 시 자동으로 음성 안내를 재생해야 한다
  - 안내 내용: "이 면접은 채용 평가 목적으로 녹음됩니다. 동의하시면 '동의합니다'라고 말씀해주세요"
- 시스템은 화상회의 채팅으로 동의 링크를 발송해야 한다
- 시스템은 지원자의 음성 동의 ("동의합니다" 발화) 또는 채팅 링크 클릭을 감지해야 한다
- 시스템은 지원자가 동의한 경우에만 녹음을 시작해야 한다
- 시스템은 지원자가 30초 내 응답하지 않을 경우, 면접관에게 Slack 알림을 발송해야 한다
- 시스템은 지원자 동의 기록 (시간, 방법)을 법적 증거로 저장해야 한다

#### 1.3 실시간 녹음 및 전사
- 시스템은 지원자 동의 후 즉시 오디오 녹음을 시작해야 한다
- 시스템은 OpenAI Whisper를 활용하여 한국어 및 영어 실시간 전사를 수행해야 한다
- 시스템은 전사 텍스트를 5초 이내 지연으로 생성해야 한다
- 시스템은 화자 구분 (면접관 vs 지원자)을 수행하여 전사 텍스트에 포함해야 한다
- 시스템은 전사 과정 중 발생한 오류 (네트워크 불안정 등)를 면접관에게 Slack으로 알려야 한다
- 시스템은 면접 중 생성된 전사 텍스트를 실시간으로 대시보드에서 확인할 수 있도록 제공해야 한다

#### 1.4 녹음 및 전사 저장
- 시스템은 오디오 녹음 파일을 AES-256 암호화하여 저장해야 한다
- 시스템은 전사 텍스트를 타임스탬프와 함께 저장해야 한다
- 시스템은 녹음 및 전사 데이터를 면접 종료 시점부터 90일간 보관해야 한다 (기본 설정)
- 시스템은 관리자가 데이터 보관 기간을 커스터마이징할 수 있도록 설정 옵션을 제공해야 한다 (즉시 삭제 ~ 1년)
- 시스템은 지원자의 삭제 요청 시 7일 내 모든 녹음 및 전사 데이터를 완전히 삭제해야 한다

### 2. AI 기반 분석 및 요약

#### 2.1 AI 요약 리포트 자동 생성
- 시스템은 면접 종료 후 3분 이내에 AI 요약 리포트를 생성해야 한다
- AI 요약 리포트는 다음 섹션을 포함해야 한다:
  - 면접 개요 (일시, 참여자, 총 시간)
  - 지원자 강점 (구체적 사례 포함)
  - 지원자 약점 또는 개선 영역
  - 주요 경험 요약
  - 역량별 증거 (예: 리더십, 프로젝트 관리, 기술 스택)
- 시스템은 AI 요약 리포트 생성 완료 시 면접관에게 Slack 알림을 발송해야 한다
- 시스템은 AI 요약 리포트를 대시보드에서 확인할 수 있도록 제공해야 한다

#### 2.2 구조화된 평가 리포트
- 시스템은 면접 내용을 직무 요구사항 (JD)과 매핑하여 구조화된 평가 리포트를 생성해야 한다
- 평가 리포트는 다음을 포함해야 한다:
  - 각 역량별 검증 여부 (검증됨 / 부분 검증됨 / 미검증)
  - 각 역량에 대한 구체적 증거 (지원자 답변 인용)
  - 검증되지 않은 역량 목록
- 시스템은 평가 리포트를 면접관이 수정할 수 있도록 편집 가능한 형태로 제공해야 한다

#### 2.3 면접관 피드백 자동 초안 생성
- 시스템은 AI 요약을 기반으로 면접관 피드백 양식의 초안을 자동 생성해야 한다
- 시스템은 면접관이 초안을 검토하고 수정할 수 있도록 편집 인터페이스를 제공해야 한다
- 시스템은 면접관의 수정 내용을 원본 AI 요약과 구분하여 저장해야 한다
- 시스템은 면접관이 수정한 피드백을 최종 제출할 수 있는 버튼을 제공해야 한다

### 3. 실시간 추가 질문 제안

#### 3.1 지원자 답변 실시간 분석
- 시스템은 지원자의 답변을 실시간으로 분석하여 역량 검증 깊이를 평가해야 한다
- 시스템은 답변이 표면적이거나 구체적 사례가 부족한 경우를 감지해야 한다
- 시스템은 면접관이 설정한 검증 목표 역량 목록을 참조해야 한다

#### 3.2 추가 질문 제안 제공
- 시스템은 지원자 답변 종료 후 10초 이내에 추가 질문 제안을 생성해야 한다
- 시스템은 추가 질문 제안을 면접관에게 Slack 또는 대시보드를 통해 실시간으로 알려야 한다
- 추가 질문 제안은 다음 형식을 따라야 한다:
  - "이 답변으로 [역량]이 충분히 검증되지 않았습니다"
  - "제안 질문: [구체적 질문 예시]"
  - "목적: [이 질문이 검증하는 내용]"
- 시스템은 면접관이 제안을 채택했는지 여부를 추적해야 한다 (질문 제안 유용성 측정)

#### 3.3 제안 품질 피드백
- 시스템은 면접관이 제안 질문의 유용성을 평가할 수 있도록 "도움됨 / 도움안됨" 버튼을 제공해야 한다
- 시스템은 면접관 피드백을 수집하여 제안 품질을 지속적으로 개선해야 한다

### 4. 온톨로지 지식 그래프

#### 4.1 면접 데이터 구조화
- 시스템은 면접 전사 내용을 분석하여 다음 정보를 추출해야 한다:
  - 언급된 역량 (예: 프로젝트 관리, 리더십, 기술 스택)
  - 각 역량의 경험 수준 (기간, 규모, 구체성)
  - 검증된 역량 vs 미검증 역량
- 시스템은 추출된 정보를 온톨로지 노드로 구조화하여 저장해야 한다
- 시스템은 지원자 프로필, 역량 노드, 면접 기록을 연결하여 지식 그래프를 구성해야 한다

#### 4.2 역량 매핑
- 시스템은 직무 요구사항 (JD)의 역량과 지원자가 언급한 역량을 자동으로 매핑해야 한다
- 시스템은 매핑 결과를 시각화하여 대시보드에 표시해야 한다 (예: "프로젝트 관리: 검증됨 (사례 3개)")
- 시스템은 누락된 역량을 하이라이트하여 표시해야 한다

#### 4.3 조직 채용 지식 축적
- 시스템은 면접 데이터가 누적될수록 조직 전체의 채용 패턴을 학습해야 한다
- 시스템은 성공한 채용 (90일 이상 재직)과 실패한 채용 (90일 내 퇴사)의 면접 패턴을 비교 분석해야 한다
- 시스템은 학습된 패턴을 기반으로 질문 제안의 정확도를 향상시켜야 한다

### 5. Slack 통합

#### 5.1 Slack 명령어
- 시스템은 다음 Slack 명령어를 지원해야 한다:
  - `@Round [지원자명] 면접 요약해줘`: 특정 지원자의 최근 면접 요약 제공
  - `@Round [지원자명]의 [역량] 역량은?`: 특정 역량에 대한 면접 증거 제공
  - `@Round 오늘 면접 목록`: 당일 진행된 모든 면접 목록 및 요약 링크 제공
- 시스템은 Slack 명령어 응답에 관련 면접 녹음 구간 링크를 포함해야 한다

#### 5.2 Slack 알림
- 시스템은 다음 시점에 Slack 알림을 발송해야 한다:
  - 면접 시작 시: "김철수 면접 녹음 시작되었습니다"
  - 면접 종료 시: "AI 요약 리포트가 준비되었습니다 [링크]"
  - 지원자 동의 미응답 시: "지원자가 녹음 동의에 응답하지 않았습니다. 수동으로 확인해주세요"
  - 중요 순간 감지 시: "지원자가 [토픽]에 대해 민감한 발언을 했습니다 [링크]"
- 시스템은 알림 수신 채널을 채용 담당자가 설정할 수 있도록 옵션을 제공해야 한다

#### 5.3 팀 공유
- 시스템은 면접 녹음 및 요약을 Slack 채널에 공유할 수 있는 기능을 제공해야 한다
- 시스템은 공유 시 권한 설정 (참여자만 / 채용팀 전체 / 특정 사용자)을 선택할 수 있도록 해야 한다
- 시스템은 공유된 녹음에 접근한 사용자 기록을 감사 로그에 남겨야 한다

### 6. 디브리프 기능 연동

#### 6.1 디브리프 자료 생성
- 시스템은 디브리프 미팅을 위한 지원자별 비교 자료를 생성해야 한다
- 비교 자료는 다음을 포함해야 한다:
  - 각 지원자의 AI 요약 리포트
  - 역량별 검증 상태 비교표
  - 핵심 답변 인용 (지원자별)
- 시스템은 디브리프 자료를 대시보드 및 Slack에서 접근할 수 있도록 제공해야 한다

#### 6.2 증거 기반 비교
- 시스템은 디브리프 미팅 중 특정 역량에 대한 지원자별 답변을 즉시 검색할 수 있는 기능을 제공해야 한다
- 시스템은 면접 녹음의 특정 구간을 재생할 수 있는 플레이어를 제공해야 한다
- 시스템은 디브리프 미팅 결과 (최종 합의 내용)를 기록할 수 있는 인터페이스를 제공해야 한다

### 7. 권한 및 보안

#### 7.1 접근 권한 관리
- 시스템은 역할 기반 접근 제어 (RBAC)를 적용해야 한다
  - 기본 설정: 면접 참여자만 접근 가능
  - 선택 설정: 채용팀 전체 접근 가능
  - 관리자 설정: 특정 사용자 지정 접근
- 시스템은 권한 설정을 면접별로 개별 지정할 수 있어야 한다

#### 7.2 데이터 암호화 및 보안
- 시스템은 녹음 데이터를 전송 중 TLS 1.3으로 암호화해야 한다
- 시스템은 녹음 데이터를 저장 중 AES-256으로 암호화해야 한다
- 시스템은 AWS KMS를 사용하여 암호화 키를 관리해야 한다

#### 7.3 감사 로그
- 시스템은 다음 활동을 감사 로그에 기록해야 한다:
  - 녹음 파일 접근 (사용자, 시간)
  - 녹음 파일 공유 (공유자, 수신자, 시간)
  - 녹음 파일 삭제 (삭제자, 시간)
- 시스템은 비정상 접근 (예: 권한 없는 사용자의 접근 시도) 시 관리자에게 알림을 발송해야 한다

### 8. 대시보드

#### 8.1 면접 목록 및 상태
- 대시보드는 진행 중인 면접, 완료된 면접, AI 요약 생성 중인 면접을 구분하여 표시해야 한다
- 대시보드는 각 면접의 상태 (녹음 중, 전사 중, 요약 생성 중, 완료)를 실시간으로 업데이트해야 한다

#### 8.2 면접 상세 보기
- 대시보드는 특정 면접 클릭 시 다음 정보를 표시해야 한다:
  - 면접 개요 (일시, 참여자, 총 시간)
  - AI 요약 리포트
  - 구조화된 평가 리포트
  - 전사 텍스트 (타임스탬프 포함)
  - 오디오 녹음 플레이어
- 대시보드는 전사 텍스트의 특정 부분 클릭 시 해당 구간 오디오 재생을 지원해야 한다

#### 8.3 설정 및 관리
- 대시보드는 다음 설정을 제공해야 한다:
  - AI 노트테이커 활성화/비활성화
  - 자동 참여 대상 화상회의 플랫폼 선택 (Zoom, Google Meet)
  - 데이터 보관 기간 설정
  - 기본 권한 설정
  - Slack 알림 채널 설정

## Non-Goals (Out of Scope)

다음 기능들은 MVP 범위에 **포함되지 않으며**, Phase 2 이후 릴리스에서 검토합니다:

1. **Microsoft Teams, Webex 지원**: MVP는 Zoom과 Google Meet만 지원
2. **오프라인 면접 녹음**: 화상 면접만 지원하며, 대면 면접을 위한 모바일 앱 녹음 기능은 제외
3. **역량 갭 자동 감지 및 실시간 알림**: 실시간 추가 질문 제안까지만 제공하며, "협업 역량이 아직 검증되지 않았습니다" 같은 자동 갭 감지는 Phase 2
4. **면접관 편향 분석 및 코칭**: "이 면접관은 특정 배경 지원자를 일관되게 낮게 평가" 같은 편향 패턴 감지는 Phase 2
5. **지원자 간 비교 대시보드**: 자동화된 지원자 비교 및 랭킹 기능은 Phase 2
6. **면접 성공 패턴 분석 및 예측**: 온톨로지 기반 예측 모델은 Phase 2
7. **조직별 커스텀 온톨로지**: 회사마다 중요한 역량을 커스터마이징하는 기능은 Enterprise 플랜 전용, Phase 2
8. **Blind Review 옵션**: 디브리프 시 지원자 배경 정보를 숨기는 기능은 Phase 2
9. **비디오 녹화**: 오디오만 녹음하며, 비디오는 녹화하지 않음
10. **실시간 번역**: 한국어/영어 전사만 지원하며, 실시간 번역 기능은 제외

## Design Considerations

### UI/UX 원칙
- **최소 개입**: 면접관이 AI 노트테이커 존재를 거의 의식하지 않도록 자동화 우선
- **Slack 중심**: 라운드의 Conversational ATS 철학에 따라 Slack을 주요 인터페이스로 활용
- **투명성**: 지원자에게 녹음 사실을 명확히 고지하고 동의 프로세스를 간결하게 설계
- **편집 가능성**: AI 요약이 완벽하지 않을 수 있으므로 면접관이 항상 수정할 수 있도록 보장

### 주요 UI 컴포넌트
1. **대시보드 - 면접 목록**: 카드 형태로 각 면접 표시 (지원자명, 일시, 상태 배지)
2. **대시보드 - 면접 상세**: 좌측 전사 텍스트, 우측 AI 요약 리포트 (2-column 레이아웃)
3. **Slack 알림**: 인라인 요약 (3-5줄) + "자세히 보기" 링크
4. **실시간 질문 제안**: 면접 중 대시보드 우측 하단에 작은 알림 카드 표시

### 기존 디자인 시스템 활용
- 라운드의 기존 UI 컴포넌트 라이브러리를 재사용
- 색상, 타이포그래피, 버튼 스타일은 기존 ATS 디자인과 일관성 유지

## Technical Considerations

### 기술 스택 제안
- **STT 엔진**: OpenAI Whisper (자체 호스팅 또는 API)
- **LLM (요약 및 분석)**: GPT-4 또는 Claude (API)
- **화상회의 통합**: Zoom SDK, Google Meet API
- **데이터 저장**: AWS S3 (녹음 파일), PostgreSQL (메타데이터, 전사 텍스트)
- **온톨로지 그래프**: Neo4j 또는 Amazon Neptune (그래프 데이터베이스)
- **암호화**: AWS KMS (키 관리), TLS 1.3, AES-256

### 기존 시스템 통합
- **라운드 ATS 백엔드**: 면접 일정, 지원자 정보, 피드백 양식과 통합
- **Slack 통합**: 기존 Slack 봇에 새로운 명령어 추가
- **디브리프 기능**: 기존 디브리프 UI에 면접 녹음 및 요약 링크 추가

### 성능 요구사항
- **전사 지연 시간**: 5초 이내
- **요약 생성 시간**: 면접 종료 후 3분 이내
- **시스템 가용성**: 99.9% (월 43분 다운타임 허용)
- **동시 녹음 지원**: 최소 50개 면접 동시 녹음/전사 가능

### 스케일링 고려사항
- 녹음 파일 크기: 1시간 면접 기준 약 50MB (압축)
- 월 1,000회 면접 진행 시: 약 50GB 저장 공간 필요
- 오디오 스트리밍 대역폭: 최소 128kbps per 면접

### 의존성
- Zoom SDK, Google Meet API의 변경 사항에 대한 모니터링 필요
- OpenAI API 사용 제한 및 비용 모니터링
- AWS S3 스토리지 비용 최적화 (압축, 보관 기간 정책)

## Open Questions

1. **전사 정확도 허용 범위**: 한국어 전사 정확도 92-95%가 실제 면접관 만족도에 충분한지? 파일럿 테스트를 통해 검증 필요
2. **실시간 질문 제안 타이밍**: 지원자 답변 종료 후 10초가 적절한지? 너무 빠르면 면접 흐름을 방해할 수 있음
3. **중요 순간 감지 기준**: "민감한 발언"을 어떻게 정의하고 감지할지? (예: 퇴사 사유, 급여 협상, 건강 문제)
4. **AI 요약 수정 권한**: 채용 담당자도 면접관의 AI 요약을 수정할 수 있어야 하는지?
5. **화상회의 녹음 권한**: Zoom/Google Meet의 녹음 정책 변경 시 대응 방안은?
6. **데이터 삭제 요청 프로세스**: 지원자가 삭제를 요청했으나 법적 분쟁 가능성이 있을 경우 어떻게 처리할지?
7. **다국어 지원 우선순위**: 영어 외 다른 언어 (중국어, 일본어) 지원 수요가 있는지? Phase 2 우선순위 결정 필요
8. **온톨로지 그래프 복잡도**: MVP에서 온톨로지를 어느 수준까지 구현할지? (기본 역량 노드만 vs 관계 추론까지)
9. **Slack vs 대시보드 사용 비율**: 사용자가 실제로 Slack을 주로 사용할지, 대시보드를 선호할지? 파일럿 후 UX 최적화 방향 결정
10. **비용 모델 검증**: 면접당 ₩5,000 Add-on 가격이 고객 가치 대비 적정한지? 파일럿 고객 피드백 필요
