# 제품 개선 핵심 가설

> Round HR 데이터 기반 제품 개선을 위한 핵심 가설 13개.
> 25개 초기 가설 → 8개 선별 → PM 리뷰를 거쳐 13개로 재구성했다.

## 선별 원칙

1. **검증 후 행동 가능성**: "이 가설이 맞다면, 제품의 어떤 부분을 어떻게 바꿀 것인가?"에 답할 수 있어야 한다
2. **현재 데이터로 검증 가능**: 새 기능 개발 없이 기존 이벤트/데이터로 분석 시작 가능해야 한다
3. **성공 + 실패 패턴 균형**: 성공한 고객뿐 아니라 이탈한 고객의 패턴도 본다
4. **User + Account 균형**: 개인 사용자뿐 아니라 팀/계정 단위 분석을 포함한다

## 가설 ID 체계

```
H-[카테고리 약자][순번]
- A: Activation (온보딩)
- E: Engagement (핵심 기능 사용)
- R: Retention (유지)
- ACC: Account (팀/계정 단위)
- REV: Revenue (매출)
- AI: AI Feature Adoption
```

## 1. Activation (온보딩) - 2개

"첫 경험이 습관을 만든다" - 신규 고객이 Round HR의 핵심 가치를 빠르게 체험하도록 한다.

| ID | 가설 | 핵심 KPI | 우선순위 |
|----|------|---------|---------|
| **H-A01** | 첫 7일 내 플랫폼 연동 설정 완료 시 → 30일 유지율 **25% 상승** | Time to First Import, Day 30 Retention | **P0** |
| **H-A02** | 첫 7일 내 첫 공고 발행 시 → 30일 유지율 **2배 이상** | Time to First Job Posting, Day 30 Retention | **P0** |

### 검증 방법
- **H-A01**: `integration_sync_completed` 이벤트의 첫 발생 시점과 Day 30 리텐션 코호트 비교. 단, 현재 연동 완료율이 30% 미만이면 가설 자체의 적용 범위를 재검토
- **H-A02**: `job_created` + `job_published` 이벤트 기반 첫 7일 내 완료 코호트 vs 미완료 코호트 비교. ATS의 가장 기본적인 Activation 행동

### Aha Moment는 가정이 아니라 발견한다

> 기존 H-A05("3명 스테이지 이동 = Aha Moment")는 독립 가설에서 제외했다.
> Aha Moment는 H-A01, H-A02의 코호트 분석 결과에서 **데이터로 발견**해야 한다.
> 플랫폼 연동, 공고 발행, 스테이지 이동, 팀원 초대 등 다양한 Activation 행동의 리텐션 상관관계를 비교하여 가장 강력한 선행지표를 찾는다.

## 2. Engagement (핵심 기능 사용) - 3개

"실제 사용 깊이가 가치를 결정한다" - 채용 담당자의 실질적 사용량과 팀 협업이 Lock-in을 만든다.

| ID | 가설 | 핵심 KPI | 우선순위 |
|----|------|---------|---------|
| **H-E01** | 주간 지원자 처리 10명 이상 고객 → 90일 유지율 **2배** | Weekly Candidates Processed, Day 90 Retention | **P0** |
| **H-E03** | 팀 협업 기능(@멘션, 코멘트) 주 1회 이상 사용 시 → 이탈률 **40% 감소** | Weekly Collaboration Events, Churn Rate | **P1** |
| **H-E05** | 3개 이상 채용 플랫폼 통합 고객 → 처리 지원자 수 **2배** 증가, 이탈률 **40% 감소** | Integrations per Customer, Monthly Candidates Processed, Churn Rate | **P1** |

### 검증 방법
- **H-E01**: `candidate_reviewed`, `candidate_stage_moved` 등 후보자 처리 이벤트의 주간 합산 기준 코호트 분석. "실제 업무에 쓰고 있는가?"의 직접 지표
- **H-E03**: `comment_created`, `mention_sent`, `feedback_requested` 이벤트 기반 팀 협업 빈도별 이탈률 비교. B2B에서 **혼자 쓰면 이탈, 팀이 쓰면 Lock-in**
- **H-E05**: 연동 플랫폼 수(1개/2개/3개+) 기준 코호트별 비교. 단, **Selection Bias 주의** - 플랫폼을 많이 연동하는 고객이 원래 채용 활동이 활발할 수 있음

## 3. Retention (유지) - 2개

"습관과 기능 폭이 유지를 만든다" - 로그인 빈도와 기능 사용 폭으로 리텐션 선행지표를 찾는다.

| ID | 가설 | 핵심 KPI | 우선순위 |
|----|------|---------|---------|
| **H-R01** | 주 3회 이상 로그인하는 고객 → 90일 유지율 **2.5배** (습관 형성 임계점) | WAU/MAU Ratio, Login Frequency Distribution | **P0** |
| **H-R03** | 11개 핵심 기능 중 **4개 이상** 꾸준히 사용 고객 → 연간 유지율 **75% 이상** | Feature Breadth Score, Annual Retention Rate | **P1** |

### 검증 방법
- **H-R01**: `user_login` 이벤트 빈도별 코호트 분석 (주 1회/3회/5회 → 90일 리텐션). 단, 로그인 **일관성(패턴)**이 총 횟수보다 중요 - 월수금 3회 vs 월요일 3회는 다름
- **H-R03**: `feature_first_use` 이벤트 기반 Feature Breadth Score 계산. 기존 임계점 7개에서 **4개로 하향** - B2B SaaS에서는 3-4개 핵심 기능만 꾸준히 써도 충분히 Lock-in됨

### 11개 핵심 기능 목록 (H-R03 기준)
1. 공고 관리 (Job)
2. 지원자 관리 (Candidate)
3. 스테이지 이동 (Stage)
4. 면접 일정 (Interview Schedule)
5. 평가/피드백 (Feedback)
6. 플랫폼 연동 (Integration)
7. AI 스크리닝 (AI Screening)
8. 채팅/메시지 (Chat/Message)
9. 팀 협업 (@멘션, 코멘트)
10. 자동화 (Auto Action)
11. 레퍼런스 체크 (Reference Check)

## 4. Account & Lock-in (계정 단위) - 2개

"B2B는 팀이 쓸 때 비로소 정착한다" - 개인이 아닌 팀/계정 단위의 사용 패턴과 데이터 축적이 전환 비용을 높인다.

| ID | 가설 | 핵심 KPI | 우선순위 |
|----|------|---------|---------|
| **H-ACC01** | 팀원 3명 이상 활성 사용자인 계정 → 연간 유지율 **85% 이상** | Active Users per Account, Annual Account Retention | **P0** |
| **H-LI01** | 누적 지원자 데이터 500명 이상 계정 → 이탈률 **60% 감소** (Data Lock-in) | Cumulative Candidates per Account, Account Churn Rate | **P1** |

### 검증 방법
- **H-ACC01**: 계정별 월간 활성 사용자 수 기준 세그먼트 분석 (1명/2명/3명+). 팀 전체가 쓸 때 Lock-in 발생 여부 확인
- **H-LI01**: 계정별 누적 `candidate_created` 수와 이탈률 상관관계. 데이터가 쌓일수록 경쟁사 전환 비용이 증가하는지 검증

## 5. Churn 패턴 (이탈 예측) - 1개

"실패 패턴도 봐야 한다" - 성공한 고객뿐 아니라 이탈하는 고객의 조기 신호를 찾는다.

| ID | 가설 | 핵심 KPI | 우선순위 |
|----|------|---------|---------|
| **H-CH01** | 첫 14일간 지원자 0명인 고객 → 30일 이탈률 **65%** | Day 14 Candidate Count = 0, Day 30 Churn Rate | **P0** |

### 검증 방법
- 가입 후 14일 시점에서 `candidate_created` 이벤트가 0인 계정 vs 1건 이상인 계정의 30일/60일/90일 이탈률 비교
- 이탈 원인 세분화: 공고 미발행? 연동 실패? 단순 탐색 후 이탈?

### 이 가설이 중요한 이유
> 성공 패턴(H-A01, H-A02)의 **거울상**이다. Activation 가설이 "무엇이 유지를 만드는가?"를 찾는다면, Churn 가설은 "무엇이 이탈을 만드는가?"를 찾는다. 두 방향에서 동시에 분석해야 전체 그림이 보인다.

## 6. Revenue (매출) - 2개

"사용 패턴이 매출 기회를 알려준다" - 업그레이드 타이밍과 결제 구조 최적화를 데이터로 포착한다.

| ID | 가설 | 핵심 KPI | 우선순위 |
|----|------|---------|---------|
| **H-REV01** | Starter 플랜 고객이 팀원 한도(5명) 도달 시 → Growth 전환 확률 **60%**, 평균 전환 시점 **3-4개월** | Seat Limit Reached Rate, Upgrade Conversion Rate, Avg Time to Upgrade | **P1** |
| **H-REV04** | 6개월 이상 유지 고객에 연간 결제 전환 제안 시 → 수용률 **50%** | Monthly → Annual Conversion Rate | **P2** |

### 검증 방법
- **H-REV01**: `plan_limit_reached` 이벤트 발생 후 `plan_upgraded` 이벤트까지의 전환율 및 소요 시간 추적. 한도에 부딪히는 시점이 Upsell 최적 타이밍
- **H-REV04**: 6개월+ 유지 월간 결제 고객 대상 연간 전환 제안 → 수용률 추적

## 7. AI Feature Adoption - 1개 (관찰용)

"최소한의 AI 트렌드는 추적한다" - AI 기능의 본격 검증은 핵심 워크플로우 정착 이후이나, 기초 데이터는 지금부터 모은다.

| ID | 가설 | 핵심 KPI | 우선순위 |
|----|------|---------|---------|
| **H-AI01** | AI 스크리닝을 1회 이상 사용한 고객 → 90일 유지율에 **유의미한 차이가 있는가?** | AI Feature Usage (Y/N), Day 90 Retention Diff | **P2** |

### 검증 방법
- `ai_screening_used` 이벤트 1회 이상 발생 코호트 vs 미사용 코호트의 90일 리텐션 비교
- 목표 수치를 사전에 설정하지 않음 - **"차이가 있는가?"를 먼저 확인**하는 탐색적 분석
- 샘플 사이즈가 작아도 트렌드를 관찰하는 것이 목적

### AI 카테고리 축소 근거
> 기존 5개 AI 가설(H-AI01~H-AI05)에서 1개로 축소했다. AI 기능이 초기 단계인 상황에서 AI 채택률 향상 가설을 검증하기보다, 핵심 워크플로우(온보딩 → 사용 → 유지 → 매출)의 기반을 먼저 다지는 것이 우선이다. 단, 향후 AI가 차별화 포인트가 될 수 있으므로 최소한의 데이터 수집은 유지한다.

## 우선순위 요약

### P0 - 즉시 검증 (5개)

현재 데이터로 **코호트 분석**만으로 검증 가능. 비즈니스 임팩트 최대.

| ID | 가설 | 카테고리 | 분석 유형 |
|----|------|---------|----------|
| H-A01 | 플랫폼 연동 → 30일 유지율 25%↑ | Activation | 성공 패턴 |
| H-A02 | 첫 공고 발행 → 30일 유지율 2배+ | Activation | 성공 패턴 |
| H-E01 | 주간 지원자 처리 10명+ → 90일 유지율 2배 | Engagement | 성공 패턴 |
| H-R01 | 주 3회 로그인 → 90일 유지율 2.5배 | Retention | 성공 패턴 |
| H-ACC01 | 팀원 3명+ 활성 → 연간 유지율 85%+ | Account | 성공 패턴 |
| H-CH01 | 첫 14일 지원자 0명 → 30일 이탈률 65% | Churn | **실패 패턴** |

### P1 - 3개월 내 (4개)

P0 패턴 발견 후 **심화 분석 및 개입 설계**.

| ID | 가설 | 카테고리 |
|----|------|---------|
| H-E03 | 팀 협업 기능 주 1회+ → 이탈률 40%↓ | Engagement |
| H-E05 | 3개+ 플랫폼 통합 → 처리량 2배, 이탈률 40%↓ | Engagement |
| H-R03 | 4개+ 핵심 기능 → 연간 유지율 75%+ | Retention |
| H-REV01 | Starter 한도 도달 → Growth 전환 60%, 3-4개월 시점 | Revenue |
| H-LI01 | 누적 지원자 500명+ → 이탈률 60%↓ | Lock-in |

### P2 - 검증 후 실행 (3개)

P0/P1 결과에 따라 **캠페인 또는 기능 개발** 결정.

| ID | 가설 | 카테고리 |
|----|------|---------|
| H-R02 | 14일 미접속 재참여 캠페인 → 이탈률 20%↓ | Retention |
| H-REV04 | 6개월+ 연간 결제 전환 → 수용률 50% | Revenue |
| H-AI01 | AI 스크리닝 1회+ → 유지율 차이 관찰 | AI |

## 가설 간 의존관계

```
[데이터 수집 인프라] ←── 모든 가설의 기반
        ↓
  ┌─────────────────────────────────────┐
  │  P0: 성공 + 실패 패턴 동시 분석       │
  │                                       │
  │  성공 패턴:                            │
  │  H-A01, H-A02 → Activation 행동 발견  │
  │  H-E01 → 실제 사용량 기준선            │
  │  H-R01 → 습관 형성 임계점              │
  │  H-ACC01 → 팀 단위 정착 기준           │
  │                                       │
  │  실패 패턴:                            │
  │  H-CH01 → 이탈 조기 신호               │
  │                                       │
  │  ──→ Aha Moment 데이터로 발견          │
  └─────────────────────────────────────┘
        ↓ 패턴 발견 후
  ┌─────────────────────────────────────┐
  │  P1: 심화 분석 + Lock-in 검증          │
  │  H-E03 → 팀 협업 = Lock-in 확인       │
  │  H-E05 → 멀티 플랫폼 = Lock-in 확인   │
  │  H-R03 → Feature Breadth 임계점       │
  │  H-REV01 → 업그레이드 타이밍 포착      │
  │  H-LI01 → 데이터 축적 = 전환 비용 상승 │
  └─────────────────────────────────────┘
        ↓ 패턴 확인 후
  ┌─────────────────────────────────────┐
  │  P2: 개입 실행                         │
  │  H-R02 → 이탈 방지 캠페인              │
  │  H-REV04 → 결제 구조 최적화            │
  │  H-AI01 → AI 전략 방향 결정            │
  └─────────────────────────────────────┘
```

**핵심 로직**: P0에서 "성공/실패 패턴"을 동시에 발견 → P1에서 "Lock-in 요소와 업그레이드 타이밍" 확인 → P2에서 "개입 실행"

## 검증 로드맵

### Phase 1 (Week 1-2): 기초 데이터 수집 및 현황 파악
- 현재 Activation Rate 확인 (첫 7일 공고 발행율, 연동 완료율)
- 평균 팀 크기 분포 확인
- 기능별 사용률 현황 (11개 기능)
- 첫 14일 지원자 0명 비율 확인

### Phase 2 (Week 3-4): P0 코호트 분석
- 가입 시점 기준 월별 코호트 구성
- Activation 행동 완료 vs 미완료 코호트 리텐션 비교
- 팀 크기별 유지율 비교
- Churn 조기 신호 패턴 분석
- **→ Aha Moment 후보 도출**

### Phase 3 (Month 2): P1 심화 분석 및 개입 설계
- Lock-in 요소 검증 (협업, 연동, 데이터 축적)
- Feature Breadth 임계점 정밀 분석
- 업그레이드 타이밍 패턴 파악
- 재참여 캠페인 대상 세그먼트 설계

### Phase 4 (Month 3): P2 실행 및 반복
- A/B 테스트 설계 (재참여 캠페인)
- 온보딩 플로우 개선 (Aha Moment 기반)
- 연간 결제 전환 제안 실행
- AI 전략 방향 결정
