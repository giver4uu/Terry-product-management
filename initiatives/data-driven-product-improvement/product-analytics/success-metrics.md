# 성공 지표 (Success Metrics)

> 13개 핵심 가설 검증을 위한 3개월 성공 지표와 Phase별 검증 방법을 정의한다.

---

## Primary Success Metrics

### 3개월 북스타 메트릭

| 지표 | 현재 (추정) | 3개월 목표 | 측정 방법 | 관련 가설 |
|------|-----------|-----------|----------|----------|
| **Activation Rate** (7일 내 공고 발행 + 연동) | ~40% | **60%** | 온보딩 퍼널 전환율 | H-A01, H-A02 |
| **WAU/MAU Ratio** (주간/월간 활성 비율) | ~40% | **60%** | `user_login` 이벤트 기반 | H-R01 |
| **Feature Breadth** (평균 사용 기능 수) | ~3/11 | **4+/11** | `feature_first_use` 기반 `feature_breadth_score` | H-R03 |
| **Account Activation** (팀 3명+ 활성 비율) | 측정 불가 | 베이스라인 확보 → **+15%p** | `active_team_members` 기반 | H-ACC01 |
| **90일 유지율** | 베이스라인 확보 필요 | **+10%p** | 가입 코호트별 Day 90 리텐션 | 전체 |
| **Starter→Growth 전환율** | ~12% | **20%** | 업그레이드 퍼널 전환율 | H-REV01 |

### 기존 대비 변경사항
- ~~AI Trust Score~~ 제거 → **Account Activation** 추가
- Feature Breadth 목표: ~~7/11~~ → **4+/11** (현실화)
- Starter→Growth 전환율: ~~25%~~ → **20%** (현실화)

## Counter-Metrics (역지표)

> 개선이 다른 영역에 부정적 영향을 주지 않는지 모니터링한다.

| 지표 | 허용 범위 | 경고 임계값 | 설명 |
|------|----------|-----------|------|
| 페이지 로드 타임 | < 3초 | > 5초 | 이벤트 추적 코드로 인한 성능 저하 |
| API 응답 시간 (p95) | < 500ms | > 1000ms | 백엔드 이벤트 전송으로 인한 지연 |
| 에러율 | < 0.1% | > 0.5% | 이벤트 추적 코드의 버그 |
| Amplitude 월간 비용 | 현재 플랜 내 | 한도 80% | 이벤트 볼륨 증가에 따른 비용 |

## Phase별 검증 계획

### Phase 0 (Week 1-2): 인프라 + 베이스라인 확보

| 검증 항목 | 방법 | 성공 기준 | 관련 가설 |
|----------|------|----------|----------|
| 이벤트 전송 정상 작동 | Amplitude Debugger | 10개 이벤트 모두 수신 확인 | 전체 |
| 프로퍼티 정확성 | 샘플 100건 수동 검증 | null율 < 5%, 값 정확도 > 95% | 전체 |
| 성능 영향 없음 | APM 모니터링 | API p95 < 500ms 유지 | 역지표 |
| 사용자 식별 정확성 | Amplitude User Lookup | user_id 매칭율 > 95% | 전체 |
| 기초 데이터 현황 파악 | Amplitude 차트 | 현재 연동 완료율, 공고 발행율, 팀 크기 분포 확인 | H-A01, H-A02, H-ACC01 |

### Phase 1 (Week 3-6): P0 가설 코호트 분석

| 검증 항목 | 방법 | 성공 기준 | 관련 가설 |
|----------|------|----------|----------|
| 플랫폼 연동 → 유지율 상관 | 코호트 분석 | 연동 완료 코호트의 30일 유지율이 유의미하게 높음 | H-A01 |
| 공고 발행 → 유지율 상관 | 코호트 분석 | 7일 내 발행 코호트의 30일 유지율이 유의미하게 높음 | H-A02 |
| 주간 처리량 → 유지율 상관 | 코호트 분석 | 주간 10명+ 처리 코호트의 90일 유지율 차이 확인 | H-E01 |
| 로그인 빈도 → 유지율 상관 | 코호트 분석 | 주 3회+ 코호트의 유지율이 유의미하게 높음 | H-R01 |
| 팀 크기 → 유지율 상관 | 계정 단위 분석 | 3명+ 활성 계정의 유지율 차이 확인 | H-ACC01 |
| 이탈 조기 신호 확인 | 코호트 분석 | 14일 지원자 0명 계정의 30일 이탈률 > 50% | H-CH01 |
| Aha Moment 발견 | 다변량 분석 | Activation 행동 중 가장 강한 리텐션 상관 행동 식별 | H-A01, H-A02 |
| 이벤트 볼륨 | Amplitude 대시보드 | 일일 5,000+ events | 전체 |

### Phase 2 (Week 7-10): P1/P2 가설 검증

| 검증 항목 | 방법 | 성공 기준 | 관련 가설 |
|----------|------|----------|----------|

## 측정 주기

### Daily
- 이벤트 전송 볼륨 (정상 범위 내 확인)
- 에러율/실패율 모니터링
- 주요 퍼널 전환율 (온보딩, 업그레이드)

### Weekly
- 이벤트 프로퍼티 null율 체크
- WAU/MAU 트렌드
- 주간 처리량/협업 이벤트 집계
- Health Score 분포 확인

### Monthly
- 13개 가설 검증 진행 현황
- 북스타 메트릭 트렌드 분석
- 코호트별 리텐션 커브 업데이트
- Amplitude 비용 검토
- 다음 달 우선순위 조정

## Amplitude 대시보드 구성 계획

### 기본 차트 8개 (Phase 0)
1. DAU/WAU/MAU 트렌드
2. 이벤트별 일일 발생 횟수
3. 신규 가입 → 첫 공고 게시 시간 분포
4. 로그인 빈도 분포 (일/주/월)
5. 플랫폼별 연동 현황
6. 계정당 활성 사용자 수 분포
7. 첫 14일 내 후보자 등록 여부 분포
8. Feature Breadth Score 분포

### 핵심 퍼널 4개 (Phase 0)
1. 온보딩 퍼널 (7단계) — H-A01, H-A02, H-CH01
2. 주간 워크플로우 퍼널 (6단계) — H-E01, H-E03
3. Account 활성화 퍼널 (6단계) — H-ACC01, H-LI01, H-E05
4. 업그레이드 퍼널 (6단계) — H-REV01, H-REV04

### 코호트 정의 (Phase 1)
- **Power Users**: 주 5회+ 로그인, 4개+ 기능 사용
- **At-Risk**: 14일+ 미접속 또는 Feature Breadth < 2
- **Team Champions**: 계정 내 활성 3명+, 주간 협업 이벤트 발생
- **Upgrade Ready**: Starter 플랜, 팀원 한도 80%+ 도달
